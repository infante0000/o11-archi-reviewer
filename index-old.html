Last-Hour Cram Sheet

One page. Read this right before you walk in. Use Print for a paper copy (fits one sheet).

The 3 Layers (top → bottom)

  • END-USER UI + Processes. Talks to users.
  • CORE APIs + Core services. Business concepts.
  • FOUNDATION Integration services + NFRs. Talks to outside world.

Rule: lower layers never call upward. External consumers hit Core (APIs); external producers/DBs sit below Foundation.

Naming Suffixes

  • Core: _CS Core Service · _BL Business Logic · _CW Core Widgets · _Eng Calc Engine · _Sync Sync module · _API exposes API externally
  • Foundation: _Lib generic library · _IS Integration Service · _Drv Driver · _Th Theme · _Pat UI Patterns · _Plug Plugin
  • Add M for mobile-only: _MTh, _MPat

Integration Pattern

  • Flow: Consumer → Core Service (_CS) → Integration Service (_IS) → External System
  • Direct Integration: always fresh, but more hits/latency/APIs
  • Local Replica: less external impact, simpler API, mashup-friendly — but needs a sync strategy
  • Transparency Service: use when >1 external source for the same concept; IS picks the right Driver

Sync Patterns (Local Replica)

  • Batch Sync: async, scheduled, simple, but data can be stale
  • Real-time Sync: pub/sub or ESB, always fresh, but needs external system callback support
  • Decide by asking: OK to replicate? Do summary fields change often? Is detail reused a lot? >1 source for same concept? (→ Transparency)

Module Elements by Layer

  • Foundation: Data, Logic, Interface (no Processes)
  • Core: Data (read-only entities), Logic (CRUD/audit/roles), Interface (business blocks), Processes (async)
  • End-User: Data (aux/temp), Logic (use-case specific), Interface (role-based UI), Processes (workflows)

Validation Rules — Modules

  • Rule 1 — No Upward References
  • Rule 2 — No Side References between End-User modules (screen refs OK)
  • Rule 3 — No Circular References
  • Extra: avoid exposing Core-type entities from Foundation

Validation Rules — Applications

  • Rule 1 — Layer your modules correctly
  • Rule 2 — Layer applications correctly (an app's layer = its highest-layer module)
  • Rule 3 — Don't mix owners (split module / promote ownership)
  • Rule 4 — Don't mix sponsors (each line-of-business deploys independently)

Refactoring Priority Order

  1. Upper/side refs to End-user modules
  2. Direct cycles between Core modules
  3. Upper refs to Core modules
  4. Direct cycles between Foundation modules

Within any tier: fix the module with the most consumers first. Freeze feature dev on modules being refactored.

Extract a Core — 6 steps

Identify consumed elements → Clone module → Rename original to _CS, clone keeps original name → Keep consumed elements + deps in new Core → Remove those from the clone → Reference the Core. (Consumers/URLs untouched.)

Moving Elements

  • Safe to move freely: Actions, Structures, Blocks, Screens
  • Watch out for: Timers (schedule), Site Properties (effective value), Web References (endpoint/URL), Roles (user assignments)
  • BPT process: copy → untrigger old (let in-flight finish) → deploy everywhere → delete old once drained
  • Entities via Refactor tool: copy to target → run Refactor → full compile (no ref changes yet) → update reference → publish

Mobile Architecture

  • 3 pillars: Lazy Load, Local Storage, Avoid Server Calls
  • Screen events order: OnInitialize → OnReady → OnRender → OnAfterFetch → (OnParametersChanged for blocks) → OnDestroy
  • Never fetch data in event handlers directly — prefer on-demand/parallel Aggregates
  • Local storage: denormalize, keep only needed attributes, avoid mirroring server schema
  • CSS lives in the Theme only — never in Blocks/Screens
  • Use SSL Pinning to stop man-in-the-middle attacks

Style Guides

  • Specialize (reference base theme): inherits updates, slightly slower
  • Clone (copy base theme): best performance, no auto-updates
  • Build your own: when nothing existing fits
  • Sharing: Independent Apps (own menu+login) · Intranet (own menu, same login) · Portal (common menu+login)

Process & Odds & Ends

  • Design process: Disclose → Organize → Assemble
  • ISO 25010:2011 = ISO 9126:1991 + Security + Compatibility
  • Modular programming benefits: Abstraction, Isolation, Complexity mgmt, Teamwork
